# TacticsGPT — Phase 1

> *"What if a language model could think like a football analyst?"*

This is the story of how TacticsGPT is built from scratch — from a raw text dump of match analyses all the way to a model that can answer questions about football tactics in plain, coherent language. No pre-trained weights borrowed from the internet. No fine-tuning a giant model you don't understand. Just you, your data, a GPU on Google Colab, and a three-stage training pipeline built cell by cell.

---

## The Big Picture

Training TacticsGPT follows three chapters:

```
Raw Text  →  [Pretrain]  →  Base GPT
                                ↓
            SFT Dataset  →  [SFT LoRA]  →  Instruction-following GPT
                                                    ↓
            RL Prompts   →  [RL LoRA]   →  TacticsGPT ✅
```

Each stage teaches the model something new. Pretraining gives it football language. Supervised fine-tuning (SFT) gives it the format of a good answer. Reinforcement learning (RL) gives it the judgment to produce *better* answers over time.

## Chapter 1 — The Corpus: Teaching the Model Football Language

Before a model can answer questions, it must first learn the *vocabulary* and *style* of the domain. That is what pretraining does, and it all starts with the corpus.

### Where the Data Comes From

The raw input is a plain `.txt` file of tactical match analysis articles, each separated by a header like `ARTICLE #42`. The file can be uploaded directly in Colab or placed at `data/raw_tactical_match_analysis.txt`.

### What the Cleaner Does

Raw text is messy. The cleaning pipeline in `src/build_corpus.py` does four things in order:

1. **Fixes encoding rot.** Garbled characters like `â€™` are signs of a UTF-8 file that was read as Windows-1252. The cleaner detects this pattern and repairs the encoding before touching anything else.

2. **Normalises punctuation.** Curly quotes, em-dashes, and non-breaking spaces are all replaced with their plain ASCII equivalents. This keeps the tokenizer from treating the same word as two different tokens just because of a quote style.

3. **Strips wrapper noise.** Lines like `TACTICAL MATCH ANALYSIS ARTICLES`, `Generated by ...`, or rows of `====` are metadata, not training content. They are removed.

4. **Chunks long documents.** Articles that run for thousands of words are split into smaller documents of roughly 9,000 characters (hard stop at 14,000). This prevents the model from learning one enormous blob as a single "article" and instead sees many coherent, self-contained pieces of tactical writing.

The result is `data/tactics_corpus.txt`, a clean file of training documents separated by blank lines. Only documents longer than 400 characters make the cut — short fragments are discarded.

## Chapter 2 — The Tokenizer: Building the Model's Alphabet

A language model does not read words. It reads *token IDs* — integers that represent sub-word pieces. Before any neural network can run, we need a tokenizer that maps text to these integers and back.

### Training the Tokenizer

Cell 4 trains a **Byte-Pair Encoding (BPE)** tokenizer using the HuggingFace `tokenizers` library. BPE works by starting with individual characters and repeatedly merging the most common adjacent pairs until a target vocabulary size is reached.

The tokenizer is trained on the same `tactics_corpus.txt` that will be used for pretraining. This matters: a tokenizer trained on football writing will have clean, efficient representations for words like *pressing*, *high-block*, *pivot*, or *4-3-3* rather than splitting them into awkward fragments.

Key parameters:
- **Vocabulary size:** 8,000 tokens — large enough for rich tactical language, small enough for a compact model.
- **Special tokens:** `<bos>` (beginning of sequence), `<eos>` (end of sequence), `<pad>`, and `<unk>`.
- **Output:** `checkpoints/tokenizer/tokenizer.json`

## Chapter 3 — Pretraining: Reading Everything

This is the longest stage. The model reads the entire cleaned corpus, over and over, and learns to predict the next token. It is learning nothing about question answering yet — it is just absorbing the patterns, style, and structure of tactical football writing.

### The Model Architecture

TacticsGPT is a **decoder-only Transformer** — the same family as GPT. It is intentionally small so that it trains in reasonable time on a free Colab GPU:

| Hyperparameter | Value |
|---|---|
| Vocabulary size | 8,000 |
| Context length | 256 tokens |
| Transformer layers | 4 |
| Attention heads | 4 |
| Model dimension (d_model) | 256 |
| FFN hidden size | 1,024 |
| Dropout | 0.1 |

Each layer is a standard **pre-norm Transformer block**: LayerNorm → Causal Self-Attention → residual, then LayerNorm → Feed-Forward → residual. The token embedding and the final output projection (`lm_head`) share their weights, which is a common trick that reduces parameters without hurting quality.

### The Training Loop

The pretraining script (`src/train_pretrain.py`) runs with these settings by default:

- **30 epochs** over the full corpus
- **Batch size 16**, with **gradient accumulation over 2 steps** (effective batch of 32)
- **Learning rate 3e-4** with a 500-step linear warmup
- **Sliding window** over documents with stride 128, so long documents generate multiple overlapping training windows
- **5% of data** held out as a validation set
- **Mixed precision (AMP)** on CUDA for speed
- Optional W&B tracking with `--wandb_project`, logging train loss, LR schedule, validation loss, and validation perplexity
- Checkpoint saved every 250 steps; best validation loss checkpoint saved separately

The script resumes automatically from the latest `checkpoints/pretrain/checkpoint_step_*.pt`. If Colab disconnects mid-run, just re-run the cell — it picks up where it left off.

> **Out of GPU memory?** Reduce `--batch_size` to `8` or `4` and raise `--grad_accum_steps` to compensate.

## Chapter 4 — Supervised Fine-Tuning: Learning to Answer

The second stage teaches the model *format*. We provide a dataset of `(instruction, response)` pairs and train the model to produce the response given the instruction. The prompt format is:

```
### Instruction:
How should a 4-4-2 defend central spaces against a 4-2-3-1?

### Response:
The 4-4-2 should...
```

### LoRA: Fine-Tuning Without Rewriting Everything

Instead of updating all the model's weights — which is expensive and risks destroying what was learned during pretraining — SFT uses **LoRA (Low-Rank Adaptation)**. LoRA freezes the original weights and injects small trainable matrices at each attention layer. During fine-tuning, only these tiny adapter matrices are updated. They are saved separately in `checkpoints/sft/`.

This has two big advantages:
- **Speed:** far fewer parameters to update.
- **Safety:** the pretrained base is never overwritten. You can always go back to it.

### The SFT Dataset

Upload a `.jsonl` file where each line is:

```json
{"instruction": "What pressing triggers should a team use against short build-up?", "response": "Against short build-up..."}
```

The dataset is saved to `data/sft_dataset.jsonl`.

### SFT Training Settings

The trainer is `src/train_sft_lora.py`. A short smoke-test run is provided (Cell 7.3) to verify everything loads correctly before committing to a full run (Cell 7.5). Key defaults:

- AdamW optimiser, low learning rate suitable for fine-tuning
- Only the LoRA adapter weights have `requires_grad = True`
- Resumes from the latest `checkpoints/sft/sft_step_*.pt`
- Optional validation split with periodic eval loss/perplexity logging to local JSONL and W&B

## Chapter 5 — Reinforcement Learning: Developing Judgment

The third and final stage is where TacticsGPT learns to be *good* rather than just *well-formatted*. SFT showed it what answers look like. RL teaches it to prefer better answers over worse ones.

### The Reward Function

The RL trainer uses **GRPO-style group-relative advantage**. For each prompt, the model generates a small group of candidate responses (typically 2). Each response is scored by a reward function that checks:

- Does the response use relevant tactical keywords from the reference dataset?
- Is the formation mentioned correctly (e.g., "4-4-2", "4-2-3-1")?
- Is the response coherent in length and structure?

Responses within a group are compared to each other. A response that scores above the group average gets a positive advantage (reinforced). One that scores below gets a negative advantage (suppressed). This relative comparison is more stable than comparing against a fixed baseline.

The RL prompts live in `data/rl_prompts.jsonl`, built from the instruction fields of the SFT dataset.

### RL Training Settings (Cell 10)

| Parameter | Default | What it controls |
|---|---|---|
| `TOTAL_STEPS` | 1,000 | How long to train |
| `--lr` | 5e-5 | Learning rate (lower than SFT) |
| `--group_size` | 2 | Responses sampled per prompt per step |
| `--temperature` | 0.75 | Sampling temperature during training |
| `--eval_every` | 25 | How often to run a reward evaluation |
| `--top_k_best` | 3 | How many top checkpoints to keep |
| `--save_every` | 25 | How often to save a recovery checkpoint |

Increasing `TOTAL_STEPS` and rerunning the cell continues training — the script resumes from the latest `rl_step_*.pt` automatically.

The RL trainer writes `checkpoints/rl/metrics.jsonl` and, when W&B is enabled, logs policy loss, LR, train reward, eval reward, best eval reward, and GRPO reward improvement percentage.

## Frequently Asked Questions

**What if I run out of GPU memory during pretraining?**
Reduce `--batch_size` to `8` then `4`. Raise `--grad_accum_steps` to `4` to maintain the effective batch size of 32.

**Can I use a different model size?**
Yes. Pass `--d_model 384 --n_heads 6 --n_layers 6` to the pretrain script for a larger model. Make sure your GPU has enough VRAM.

**Can I test the SFT model without doing RL?**
Yes. In Cells 11 or 12, set `ADAPTER_DIR = "checkpoints/sft"`. The model and adapter will load from the SFT checkpoints instead.

**How do I intentionally start over?**
Set `START_FROM_SCRATCH = True` in Cell 1 and rerun. This deletes the corpus, tokenizer, and pretrain checkpoints. You will need to go through all stages again from the beginning.

**What is the expected training time?**
On a Colab T4 GPU: tokenizer trains in seconds, pretraining takes several hours for 30 epochs, SFT and RL each take minutes to hours depending on dataset size and step count.
